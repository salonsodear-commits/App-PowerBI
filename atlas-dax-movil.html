🟢 Contexto de Fila

"¿En qué fila estoy parado ahora mismo?" El contexto que te da acceso a los valores de una fila concreta.

1 · Explicación simple

El contexto de fila es la idea de "estar parado en una fila concreta de una tabla". Cuando estás en esa fila, puedes leer el valor de cualquier columna de esa fila. Es exactamente lo que ocurre en Excel cuando escribes una fórmula en la celda D2 que usa B2*C2: estás "en la fila 2".

2 · Explicación conceptual

El contexto de fila aparece en dos situaciones, y solo en dos:

  • Columnas calculadas: se evalúan fila por fila para toda la tabla. Cada fila trae su propio contexto de fila automáticamente.
  • Funciones iteradoras (las que terminan en X: SUMX, AVERAGEX, RANKX…, y también FILTER): recorren una tabla fila a fila y, en cada paso, abren un contexto de fila.
🎭 Analogía: El dedo sobre el renglón

Imagina que lees una tabla impresa con el dedo apoyado en un renglón. Mientras tu dedo está en la fila de "Juan", puedes leer su sueldo, su área, su antigüedad. El contexto de fila es ese dedo: te da acceso a los valores de la fila donde estás parado, una a la vez.

3 · Explicación técnica

El contexto de fila NO filtra el modelo. Esta es la confusión número uno. Estar parado en la fila de "Juan" no filtra automáticamente el resto del modelo a Juan. Solo te da acceso a leer las columnas de esa fila.

DAXMargen = Ventas[Precio] - Ventas[Costo]   // columna calculada
// En cada fila, lee Precio y Costo DE ESA FILA. Eso es contexto de fila.
DAXIngreso Total =
SUMX (
    Ventas,                              -- itera fila a fila sobre Ventas
    Ventas[Cantidad] * Ventas[Precio]    -- en cada fila lee Cantidad y Precio
)
⚙️ Qué ocurre internamente

Internamente, SUMX(Ventas, ...) hace esto:

  1. Toma la tabla Ventas ya filtrada por el contexto de filtro actual.
  2. Para cada fila, abre un contexto de fila apuntando a esa fila.
  3. Evalúa la expresión (Cantidad * Precio) leyendo los valores de esa fila.
  4. Acumula el resultado y pasa a la siguiente fila.

4 · El detalle que separa intermedios de seniors ⚠️

⚠️ Contexto de fila NO recorre relaciones por sí solo

Si estás en una fila de Ventas y quieres el nombre de la categoría que vive en la tabla Productos, el contexto de fila no cruza la relación automáticamente. Necesitas RELATED para "saltar" del lado muchos al lado uno:

DAXVenta con Categoría =
SUMX (
    Ventas,
    Ventas[Cantidad] * RELATED ( Productos[Precio] )   -- RELATED cruza la relación
)

Sin RELATED, el contexto de fila de Ventas no "ve" las columnas de Productos.

5 · Caso empresarial

Cálculo de OPEX ponderado por tipo de cambio. Tienes gastos en distintas monedas y un tipo de cambio por fila. No puedes sumar primero y convertir después; debes convertir fila a fila:

DAXOPEX en USD =
SUMX (
    Gastos,
    Gastos[Importe Local] * RELATED ( TipoCambio[Tasa USD] )
)

6 · Errores frecuentes

❌ Usar SUM cuando necesitas SUMX

SUM(Ventas[Cantidad] * Ventas[Precio]) no compila. SUM solo acepta una columna, no una expresión por fila. Necesitas SUMX porque hay una operación que debe ocurrir en cada fila antes de sumar.

❌ Creer que el contexto de fila filtra

Una columna calculada Total Cliente = SUM(Ventas[Importe]) en la tabla de clientes da el gran total para todos, no el total de ese cliente. ¿Por qué? Porque el contexto de fila no se convierte en filtro automáticamente. Para eso necesitas transición de contexto (CALCULATE).

7 · Mejores prácticas

✅ Mejores prácticas
  • Usa iteradores (X) solo cuando necesitas operar fila a fila. Si la columna ya tiene el valor listo, SUM es más barato.
  • Prefiere RELATED dentro de iteradores a crear columnas calculadas que dupliquen datos.
  • Recuerda: contexto de fila = acceso a valores, contexto de filtro = qué filas existen. Son cosas distintas.
✏️ Predice el resultado

Tabla Ventas: 3 filas → (Cant 2, Precio 10), (Cant 1, Precio 50), (Cant 4, Precio 5).

¿Cuánto da SUMX(Ventas, Ventas[Cantidad] * Ventas[Precio])?

Ver solución paso a paso

Fila 1: 2×10 = 20 · Fila 2: 1×50 = 50 · Fila 3: 4×5 = 20. SUMX acumula: 20+50+20 = 90. Nota que SUM(Cant)×SUM(Precio) daría 7×65 = 455, ¡completamente distinto! Ese es el sentido del iterador: multiplica dentro de cada fila.

El contexto de filtro es la comanda que llega a la cocina: "mesa 4, sin gluten, para 2 personas". El cocinero (la medida) solo cocina lo que la comanda especifica. Cada celda de tu informe genera una comanda distinta, y por eso la misma medida da números diferentes en cada celda. Una sola receta, miles de comandas.

3 · Explicación técnica: filtra el MODELO, no la medida

El contexto de filtro actúa antes de que tu medida se ejecute: filtra físicamente las tablas. Cuando SUM(Ventas[Importe]) corre en la celda "Marzo", el motor ya recortó Ventas a solo las filas de marzo. Tú no escribiste ningún filtro: lo puso el visual.

⚙️ Qué ocurre internamente: Secuencia exacta en una celda
  1. El visual declara sus coordenadas: {Mes=Marzo, Cliente=A}.
  2. El motor aplica esos filtros sobre las dimensiones (Calendario, Clientes).
  3. Esos filtros se propagan por las relaciones hasta la tabla de hechos Ventas, recortándola.
  4. La medida se evalúa sobre esa tabla ya recortada.
  5. El número resultante se pinta en la celda.

4 · El gran total y el problema de "no hay filtro"

⚠️ La fila Total tiene un contexto de filtro distinto

En la fila Total de una matriz, no hay filtro de Cliente ni de Mes. El contexto de filtro está "vacío" en esas columnas, así que la medida ve todas las filas. Por eso un porcentaje mal hecho da 100% en cada fila pero el total no cuadra: cada celda vive en un contexto distinto. Entender esto es clave para los simuladores en matriz.

5 · Caso empresarial: % sobre el total

Para calcular qué porcentaje del EBITDA total aporta cada unidad de negocio, necesitas dos contextos a la vez: el de la fila (esta UN) y uno sin filtro de UN (el total). Ahí entra CALCULATE + ALL:

DAX% sobre Total =
DIVIDE (
    [EBITDA],                                    -- contexto de la fila (esta UN)
    CALCULATE ( [EBITDA], ALL ( UnidadNegocio ) ) -- contexto sin filtro de UN = total
)

6 · Errores frecuentes

❌ Confundir contexto de filtro con contexto de fila

Son ortogonales. El contexto de fila te deja leer valores de una fila; el de filtro decide qué filas existen para agregar. Un iterador abre contexto de fila pero NO cambia el de filtro (hasta que metes CALCULATE en medio → transición).

❌ Asumir que un slicer "ya filtró la medida"

El slicer filtra el modelo, no "la medida". Si dentro usas ALL(), borras ese filtro y el slicer deja de tener efecto. Saber qué filtros conserva o borra cada función es el núcleo del módulo CALCULATE.

7 · Mejores prácticas

✅ Mejores prácticas
  • Antes de escribir una medida, pregúntate: "¿en qué contexto de filtro se va a evaluar?" (¿celda de matriz? ¿total? ¿con qué slicers?).
  • Para "romper" el contexto usa funciones explícitas (ALL, REMOVEFILTERS, ALLSELECTED) y entiende exactamente cuál necesitas.
  • Diseña dimensiones limpias: el contexto de filtro funciona mejor con un buen star schema.
✏️ Predice el resultado

Matriz con Clientes en filas. Medida Ventas = SUM(Ventas[Importe]). Cliente A tiene 100, B tiene 200, C tiene 300.

¿Qué muestra cada celda y qué muestra el Total?

Ver solución

Fila A: contexto {Cliente=A} → 100. Fila B → 200. Fila C → 300. Total: sin filtro de Cliente → ve las tres filas → 600. La misma medida da 4 números distintos porque cada celda impone un contexto de filtro diferente. Esa es toda la magia de Power BI.

💡 Por qué importa: Es el puente entre los dos contextos

Hasta ahora tienes dos mundos separados: el contexto de fila (leer valores) y el de filtro (qué filas existen). La transición de contexto es el único puente entre ellos. Entenderla explica por qué una medida dentro de SUMX "de repente" funciona, por qué una columna calculada con una medida da el valor correcto por fila, y por qué a veces tu cálculo "se filtra solo" sin que lo pidieras.

1 · Explicación simple

Transición de contexto = "convertir dónde estoy parado en un filtro". Cuando hay un contexto de fila y aparece CALCULATE (o llamas a una medida, que lleva un CALCULATE implícito), el motor toma los valores de la fila actual y los transforma en filtros sobre el modelo.

🎭 Analogía: De señalar a aislar

En el contexto de fila tu dedo señala la fila de "Cliente A" (puedes leer sus datos, pero el resto del modelo sigue completo). La transición de contexto es como decir: "ahora el restaurante entero solo cocina para Cliente A". Pasaste de señalar una fila a aislar todo el modelo a esa fila. Ese salto lo hace CALCULATE.

2 · El disparador: CALCULATE (explícito o implícito)

⚠️ Toda medida lleva un CALCULATE invisible

Cuando escribes [Ventas] (una medida) dentro de un iterador, DAX la envuelve automáticamente: se comporta como CALCULATE([Ventas]). Por eso llamar a una medida dentro de un contexto de fila SIEMPRE dispara transición de contexto, aunque no escribas CALCULATE. Esta es la fuente nº1 de resultados "mágicos" que no entiendes.

3 · El ejemplo canónico

Columna calculada en la tabla Clientes:

DAX-- ❌ SIN transición: el contexto de fila NO filtra
Total A = SUM ( Ventas[Importe] )
-- Resultado: el GRAN TOTAL en cada fila (no filtra por cliente)
DAX-- ✅ CON transición: CALCULATE convierte la fila en filtro
Total B = CALCULATE ( SUM ( Ventas[Importe] ) )
-- Resultado: el total DE ESE cliente en cada fila
⚙️ Qué ocurre internamente: Qué hace CALCULATE en Total B, paso a paso
  1. Estamos en la fila de, digamos, "Cliente A" (contexto de fila).
  2. CALCULATE detecta que hay contexto de fila activo.
  3. Lee todas las columnas de la fila actual: Clientes[ClienteID]=A, Clientes[Nombre]="A", etc.
  4. Las convierte en filtros: contexto de filtro += {Clientes[ClienteID]=A …}.
  5. Ese filtro se propaga por la relación Clientes→Ventas, recortando Ventas a las de A.
  6. SUM(Ventas[Importe]) se evalúa ya filtrado → total de A. ✔

4 · Dónde aparece en la práctica

5 · Caso empresarial: clientes por encima de su media

"¿Cuántos clientes facturan más que el promedio de facturación por cliente?" Necesitas, por cada cliente, su propio total (transición) y compararlo:

DAXClientes Sobre Media =
VAR MediaPorCliente =
    AVERAGEX ( Clientes, [Ventas] )       -- [Ventas] dispara transición: total de cada cliente
RETURN
    COUNTROWS (
        FILTER ( Clientes, [Ventas] > MediaPorCliente )  -- de nuevo transición por fila
    )

Sin transición, [Ventas] daría el gran total en cada fila y todos serían iguales. La transición es lo que hace que [Ventas] signifique "las ventas de este cliente".

6 · Errores frecuentes

❌ Performance: transición masiva

SUMX(TablaGrande, [Medida]) dispara una transición de contexto por cada fila. Sobre millones de filas, esto puede ser lentísimo. A menudo se reescribe iterando sobre una dimensión pequeña o evitando la medida dentro del iterador. Ver Performance.

❌ Transición no deseada que "ensucia" el filtro

A veces la transición añade filtros que NO querías (todas las columnas de la fila). Si solo necesitabas filtrar por una clave, usa VALUES/SELECTEDVALUE en lugar de confiar en la transición completa.

7 · Mejores prácticas

✅ Mejores prácticas
  • Cuando veas una medida llamada dentro de un iterador, di mentalmente: "aquí hay transición de contexto". Eso explica el 90% de los resultados extraños.
  • Itera sobre dimensiones pequeñas (Clientes, Productos), no sobre la tabla de hechos, cuando uses transición.
  • Si solo necesitas un valor escalar de la fila, considera capturarlo con VAR antes de entrar en CALCULATE.
✏️ El clásico que todos fallan

Columna calculada en Productos: Ventas Prod = SUM(Ventas[Importe]). Hay 3 productos. Las ventas totales del modelo son 1000.

¿Qué muestra cada fila?

Ver solución

Las tres filas muestran 1000 (el gran total), porque SUM sin CALCULATE no convierte el contexto de fila en filtro. Para obtener las ventas de cada producto necesitas CALCULATE(SUM(Ventas[Importe])) → eso dispara la transición y cada fila muestra su propio total. Recuerda esta diferencia: es la prueba de fuego de si entendiste el concepto.

💡 Por qué importa: Las relaciones deciden qué filtra a qué

El contexto de filtro nace en las dimensiones, pero el dato vive en los hechos. Las relaciones son las tuberías por donde el filtro viaja de una a otro. Si entiendes la dirección y la cardinalidad, sabes de antemano qué filtra qué; si no, vivirás peleando con números que "no responden" a los slicers.

   DIRECCIÓN SIMPLE (recomendada)          DIRECCIÓN BIDIRECCIONAL (cuidado)
   ───────────────────────────            ──────────────────────────────
       Clientes (1)                            Clientes (1)
           │  el filtro baja                       │ ▲  el filtro va y vuelve
           ▼  hacia los hechos                     ▼ │

   al lado "*". Predecible y rápido.        Útil a veces, pero crea ambigüedad
                                            y riesgo de performance.
⚙️ Qué ocurre internamente: Propagación paso a paso

Al seleccionar "Cliente A" en un slicer:

  1. El filtro se aplica sobre la tabla Clientes (queda 1 fila visible: A).
  2. Por la relación 1:* el filtro desciende a Ventas: solo filas con ClienteID = A.
  3. Cualquier medida sobre Ventas se evalúa ya sobre ese subconjunto.

El filtro no sube de Ventas a Productos por sí solo (son dos dimensiones distintas que solo se "ven" a través de la tabla de hechos, y solo si la dirección lo permite).

3 · Por qué la dirección simple es la mejor práctica

⚠️ La bidireccional es tentadora y traicionera

Activar filtrado bidireccional "para que todo filtre todo" parece cómodo, pero genera:

  • Ambigüedad: el motor puede tener varios caminos para propagar un filtro y no saber cuál usar.
  • Performance: propagación más cara.
  • Resultados inesperados: filtros que llegan a tablas donde no los esperabas.

Solución senior: deja todo en simple y, cuando puntualmente necesites que un filtro suba, hazlo dentro de una medida con CROSSFILTER o TREATAS, no a nivel de modelo.

4 · Relaciones inactivas y USERELATIONSHIP

Puedes tener varias relaciones entre dos tablas (ej. Ventas[FechaPedido] y Ventas[FechaEntrega] hacia Calendario), pero solo una activa. Las demás se activan puntualmente:

DAXVentas por Fecha Entrega =
CALCULATE (
    [Ventas],
    USERELATIONSHIP ( Ventas[FechaEntrega], Calendario[Fecha] )
)

5 · Caso empresarial: aging de cobranzas

Una tabla Facturas con FechaEmisión y FechaVencimiento, ambas hacia el calendario. El aging (antigüedad de deuda) requiere medir por fecha de vencimiento aunque la relación activa sea por emisión → USERELATIONSHIP.

✅ Reglas de oro de modelado relacional
  • Dimensión en el lado "1", hechos en el lado "*". Siempre.
  • Dirección de filtro simple por defecto.
  • Evita relaciones *:* salvo que sepas exactamente por qué; suele resolverse con una tabla puente.
  • Una sola tabla Calendario marcada como tabla de fechas para toda la inteligencia de tiempo.
✏️ Diagnóstico

Un slicer de "Categoría de Producto" no afecta a una medida que cuenta clientes. ¿Por qué podría ser?

Ver solución

El filtro de Categoría baja de Productos a Ventas (1:*), pero para llegar a Clientes tendría que subir de Ventas a Clientes — y la dirección simple no lo permite. La medida de clientes "no ve" el filtro de categoría. Soluciones: medir clientes que compraron esa categoría con CALCULATE(DISTINCTCOUNT(Ventas[ClienteID])) (cuenta sobre la tabla de hechos ya filtrada), o usar CROSSFILTER. La elección correcta depende de la pregunta de negocio.

  ┌──────────────────────────────────────────────────────────────────────┐

  │    (coordenadas de la celda + slicers + filtros de página)             │

  │    (VertiPaq) y combina los resultados                                 │
  ├──────────────────────────────────────────────────────────────────────┤
  │ 8. El número se PINTA en la celda. Siguiente celda → vuelve al paso 1  │

Aplicado a una medida real, narrado

VAR MargenYTD     = CALCULATE ( [Margen], DATESYTD ( Calendario[Fecha] ) )
VAR MargenYTD_AA  = CALCULATE ( [Margen], DATESYTD ( SAMEPERIODLASTYEAR ( Calendario[Fecha] ) ) )
RETURN
    DIVIDE ( MargenYTD - MargenYTD_AA, MargenYTD_AA )
⚙️ Qué ocurre internamente: Narración en la celda "Marzo 2026 / Región Norte"
  1. Contexto inicial: {Mes ≤ implícito por la celda = Marzo, Año=2026, Región=Norte} desde el visual y slicers.
  2. VAR MargenYTD: CALCULATE toma DATESYTD, que reemplaza el filtro de fecha por "1-ene-2026 … 31-mar-2026". Región=Norte se conserva.
  3. El nuevo filtro de fechas se propaga por la relación Calendario→Ventas; queda el Q1 de la región Norte.
  4. [Margen] se evalúa sobre ese subconjunto → margen acumulado del trimestre.
  5. VAR MargenYTD_AA: igual, pero SAMEPERIODLASTYEAR desplaza el rango a 2025 antes del YTD.
  6. RETURN: DIVIDE calcula la variación y evita el error si el año anterior fue 0.
  7. Se pinta el % en la celda. La siguiente celda (Abril) reinicia todo el proceso con su propio contexto.
✅ Cómo usar este flujo para depurar
  1. Identifica el contexto inicial: ¿en qué celda estás? ¿qué slicers hay?
  2. Localiza cada CALCULATE y pregúntate: ¿qué filtro agrega, reemplaza o conserva?
  3. Marca los iteradores y recuerda que las medidas dentro disparan transición.
  4. Sigue la propagación por las relaciones hasta los hechos.
  5. Si el número no cuadra, casi siempre el fallo está en el paso 2 (qué hace CALCULATE) o en el 3 (la propagación no llega).

💡 Por qué importa: CALCULATE es la única función que cambia el contexto de filtro

Todo lo demás en DAX lee el contexto de filtro. CALCULATE es la única que lo MODIFICA. Como el contexto de filtro es lo que decide qué datos entran en cada cálculo, controlar ese contexto = controlar Power BI. Por eso el 90% de las medidas no triviales tienen un CALCULATE en su núcleo: comparativos, % del total, time intelligence, simuladores, KPIs… todo pasa por aquí.

1 · Explicación simple

CALCULATE hace dos cosas: (1) aplica los filtros que le pasas y (2) calcula la expresión dentro de ese nuevo contexto. Su forma es:

DAXCALCULATE ( <expresión a calcular> , <filtro 1> , <filtro 2> , … )

Lee "CALCULATE" mentalmente como: "calcula ESTO, pero cambiando el contexto ASÍ".

2 · Explicación conceptual: modificar el contexto

🎭 Analogía: CALCULATE es el chef que reescribe la comanda

Recuerda la cocina: el contexto de filtro es la comanda. Una medida normal cocina lo que la comanda dice. CALCULATE intercepta la comanda y la reescribe antes de cocinar. Puede tachar líneas ("ignora el filtro de año"), añadir líneas ("solo productos premium") o reemplazar líneas ("cambia 'este mes' por 'todo el año'"). El cocinero entonces prepara el plato según la comanda corregida, no la original.

4 · Regla de oro: ¿qué hace con CADA filtro? ⚠️

⚠️ La regla que debes interiorizar

Cuando CALCULATE recibe un filtro sobre la columna X:

  • Reemplaza cualquier filtro previo que existiera sobre esa misma columna X.
  • Conserva todos los filtros sobre las demás columnas.

Ejemplo: la celda tiene {Año=2026, Región=Norte}. Escribes CALCULATE([V], Calendario[Año]=2025). Resultado: el filtro de Año pasa a 2025 (reemplazado), pero Región=Norte se conserva. Esta regla explica por qué SAMEPERIODLASTYEAR solo cambia la fecha y respeta los demás slicers.

⚙️ Qué ocurre internamente: Cómo "piensa" el motor cuando encuentra CALCULATE

Secuencia interna exacta:

  1. Detecta contexto de fila. Si CALCULATE se ejecuta dentro de un contexto de fila (iterador, columna calculada), primero hace transición de contexto: convierte la fila actual en filtros. (Esto pasa ANTES de aplicar tus filtros explícitos.)
  2. Evalúa los argumentos de filtro de izquierda a derecha. Cada uno produce una tabla de valores permitidos para una o más columnas.
  3. Fusiona esos filtros con el contexto existente aplicando la regla de oro (reemplaza por columna, conserva el resto).
  4. Construye el nuevo contexto de filtro y lo propaga por las relaciones.
  5. Evalúa la expresión (primer argumento) dentro de ese contexto ya modificado.
  6. Al terminar, descarta el contexto modificado: fuera de CALCULATE todo vuelve a la normalidad.

Detalle clave de orden: la transición ocurre primero, luego tus filtros. Y tus filtros explícitos pueden sobrescribir incluso lo que trajo la transición.

7 · El error mental más común ⚠️

❌ Creer que CALCULATE "suma" o "cuenta"

CALCULATE no agrega nada por sí mismo. No suma, no cuenta, no promedia. Solo cambia el contexto y deja que la expresión interior haga el cálculo. CALCULATE(123) devuelve 123. Toda la potencia está en los filtros, no en CALCULATE en sí. Internalizar esto evita el 50% de la confusión.

Modelo de ejemplo para todos los niveles: Ventas(Fecha, ClienteID, ProductoID, Importe) · Clientes · Productos(Tipo: Premium/Estándar) · Calendario. Medida base: Ventas = SUM(Ventas[Importe]).

1 CALCULATE simple — añadir un filtro fijo 🟢▸

Calcular ventas solo de productos premium, sin importar lo que haya en el visual.

DAXVentas Premium = CALCULATE ( [Ventas], Productos[Tipo] = "Premium" )
⚙️ Qué ocurre internamente

CALCULATE agrega el filtro Tipo="Premium" al contexto existente. Si la celda ya tenía {Mes=Marzo}, ahora es {Mes=Marzo, Tipo=Premium}. La medida suma solo esa intersección.

Cuándo: KPIs de un segmento fijo (canal premium, una línea de negocio). Resultado: en cada celda, las ventas premium correspondientes a esa celda.

2 CALCULATE con varios filtros 🟢▸

Premium y de la región Norte. Varios filtros = se aplican todos (AND).

DAXVentas Premium Norte =
CALCULATE (
    [Ventas],
    Productos[Tipo] = "Premium",
    Clientes[Región] = "Norte"
)
⚠️ Es un AND, no un OR

Múltiples argumentos de filtro se combinan con AND. Para un OR sobre la misma columna usa Productos[Tipo] IN {"Premium","Gold"} o un FILTER con ||.

3 CALCULATE + FILTER — condiciones complejas 🟡▸

Ventas de pedidos grandes: importe de línea por encima de un umbral que depende de otra columna. El predicado simple no alcanza → FILTER.

DAXVentas Pedidos Grandes =
CALCULATE (
    [Ventas],
    FILTER (
        Ventas,
        Ventas[Importe] > 1000
    )
)
⚙️ Qué ocurre internamente

FILTER recorre Ventas fila a fila (contexto de fila), se queda con las que cumplen, y devuelve esa tabla como filtro a CALCULATE. Más caro que un predicado simple, pero permite lógica que mira varias columnas o usa medidas.

❌ FILTER sobre toda la tabla de hechos

Si solo filtras una columna por un valor, no uses FILTER: usa el predicado simple (nivel 1). FILTER aquí se justifica por la comparación numérica fila a fila.

4 CALCULATE + ALL — el % del total 🟡⚠️▸

ALL quita filtros. Sirve para denominadores: el total ignorando la fila actual.

DAX% del Total =
DIVIDE (
    [Ventas],
    CALCULATE ( [Ventas], ALL ( Clientes ) )
)
⚙️ Qué ocurre internamente

En la fila "Cliente A": el numerador respeta {Cliente=A} → ventas de A. El denominador, gracias a ALL(Clientes), borra el filtro de cliente → ventas de todos. El cociente es la cuota de A. Nota: ALL(Clientes) quita filtros de la tabla Clientes pero conserva otros (ej. Año del slicer).

⚠️ ALL(Tabla) vs ALL(Columna)

ALL(Clientes) quita filtros de toda la tabla; ALL(Clientes[Región]) solo de esa columna. Elige el alcance correcto según el denominador que quieras.

5 CALCULATE + REMOVEFILTERS — la versión moderna y legible 🟡▸

REMOVEFILTERS es ALL usado como removedor de filtros, pero con nombre autoexplicativo (DAX moderno).

DAX% del Total (moderno) =
DIVIDE (
    [Ventas],
    CALCULATE ( [Ventas], REMOVEFILTERS ( Clientes ) )
)
✅ Mejores prácticas

Cuando la intención es "quitar filtros", prefiere REMOVEFILTERS: hace el código legible para quien lo mantenga. Reserva ALL para cuando lo usas como tabla (ej. dentro de SUMX). Funcionalmente, como modificador de CALCULATE, son equivalentes.

6 CALCULATE + SELECTEDVALUE — leer un slicer 🟡⭐▸

SELECTEDVALUE lee la selección única de un slicer. Base de todos los simuladores.

DAXVentas Ajustadas =
VAR Ajuste = SELECTEDVALUE ( Parametro[Factor], 1 )   -- 1 = sin ajuste si nada elegido
RETURN
    [Ventas] * Ajuste
⚙️ Qué ocurre internamente

SELECTEDVALUE(Parametro[Factor], 1) devuelve el valor si hay exactamente uno seleccionado; si hay varios o ninguno, devuelve el valor por defecto (1). Es IF(HASONEVALUE(...), VALUES(...), default) en una línea. Aquí no hace falta CALCULATE para leer; sí lo necesitarás cuando el ajuste deba filtrar.

7 CALCULATE + Inteligencia del Tiempo 🔴⭐▸

Las funciones de tiempo son generadores de tablas de fechas que CALCULATE usa para reemplazar el filtro de calendario.

DAXVentas YTD       = CALCULATE ( [Ventas], DATESYTD ( Calendario[Fecha] ) )
Ventas Año Ant.  = CALCULATE ( [Ventas], SAMEPERIODLASTYEAR ( Calendario[Fecha] ) )
Var % YoY        = DIVIDE ( [Ventas] - [Ventas Año Ant.], [Ventas Año Ant.] )
⚙️ Qué ocurre internamente

DATESYTD toma las fechas visibles y devuelve "desde el 1-ene hasta la última fecha visible". CALCULATE reemplaza el filtro de fecha por ese rango y conserva los demás (región, producto). Por eso el YTD respeta tus otros slicers automáticamente. Requiere tabla calendario marcada.

8 CALCULATE + SUMX — agregaciones por fila con contexto 🔴⭐▸

Combinar el cambio de contexto (CALCULATE) con el cálculo fila a fila (SUMX). Ejemplo: forecast = sumar, por cliente, su venta base por su factor de crecimiento.

DAXForecast =
SUMX (
    VALUES ( Clientes[ClienteID] ),               -- una fila por cliente
    VAR VentaBase = [Ventas]                       -- transición: ventas de ESE cliente
    VAR Factor    = [Factor Crecimiento Cliente]   -- su factor individual
    RETURN VentaBase * Factor
)
⚙️ Qué ocurre internamente

SUMX itera por cliente. En cada fila, [Ventas] dispara transición de contexto (CALCULATE implícito): filtra a ese cliente y suma. Se multiplica por su factor y se acumula. Aquí conviven los tres conceptos: contexto de fila (SUMX), transición (la medida dentro) y modificación de contexto.

⚠️ Itera sobre la dimensión, no sobre los hechos

Usamos VALUES(Clientes[ClienteID]) (pocas filas), no Ventas (millones). Iterar la tabla de hechos con transición dentro es un antipatrón de performance.

9 Casos corporativos avanzados 🔴⭐⚠️▸
9a · Variación Real vs Presupuesto, solo cuentas de OPEX
DAXDesvío OPEX =
VAR Real =
    CALCULATE ( [Importe], Escenario[Tipo] = "Real",    PlanCuentas[Grupo] = "OPEX" )
VAR Ppto =
    CALCULATE ( [Importe], Escenario[Tipo] = "Ppto", PlanCuentas[Grupo] = "OPEX" )
RETURN
    Real - Ppto

Cada CALCULATE reemplaza el filtro de Escenario y agrega el de grupo de cuentas, conservando el resto del contexto (mes, CECO).

9b · Margen contributivo sobre lo seleccionado (ALLSELECTED)
DAX% sobre Selección =
DIVIDE ( [Margen], CALCULATE ( [Margen], ALLSELECTED ( Productos ) ) )

ALLSELECTED da el total de lo que el usuario filtró en el slicer, ignorando solo el desglose interno del visual. Distinto de ALL (que ignoraría también el slicer).

9c · Ranking de clientes que respeta filtros, con desempate
DAXRanking Cliente =
IF (
    HASONEVALUE ( Clientes[ClienteID] ),
    RANKX ( ALLSELECTED ( Clientes[ClienteID] ), [Ventas], , DESC, DENSE )
)
✅ Patrón senior

Fíjate en la receta repetida: VAR para capturar, CALCULATE para reescribir el contexto, una función de filtro (ALL/ALLSELECTED/REMOVEFILTERS) para definir el universo, y un guard (HASONEVALUE/SELECTEDVALUE) para los totales. Casi toda medida corporativa avanzada es una combinación de estas cuatro piezas.

✏️ Predice antes de desplegar

Celda "Cliente B / Marzo". Medida: CALCULATE([Ventas], ALL(Clientes), Calendario[Mes]="Marzo"). B tiene 50 en marzo; todos los clientes juntos tienen 500 en marzo.

Ver solución

500. ALL(Clientes) borra el filtro Cliente=B (ya no importa que estemos en la fila de B). Calendario[Mes]="Marzo" reemplaza/confirma el mes. Resultado: ventas de todos los clientes en marzo = 500, en todas las filas de la matriz. Un error clásico: esperar 50 porque "estoy en la fila de B" — pero ALL lo eliminó.

1 · Cómo leer una medida: de dentro hacia afuera

DAX se evalúa desde el paréntesis más interno hacia afuera, y las variables de arriba a abajo. Lee igual:

DAXCrecimiento Clientes Clave =
VAR Umbral      = 100000                                    -- (1) un número
VAR TopClientes =                                          -- (2) tabla de clientes grandes
    FILTER ( VALUES ( Clientes[ClienteID] ), [Ventas] > Umbral )
VAR VentaTop    = CALCULATE ( [Ventas], TopClientes )       -- (3) ventas de esos clientes
VAR VentaTopAA  =                                          -- (4) lo mismo, año anterior
    CALCULATE ( [Ventas], TopClientes, SAMEPERIODLASTYEAR ( Calendario[Fecha] ) )
RETURN
    DIVIDE ( VentaTop - VentaTopAA, VentaTopAA )            -- (5) variación %
⚙️ Qué ocurre internamente: Lectura narrada
  1. Umbral = 100.000. Constante.
  2. TopClientes: recorre clientes; [Ventas] dispara transición → ventas de cada cliente; conserva los que superan el umbral. Es una tabla.
  3. VentaTop: CALCULATE usa esa tabla como filtro → ventas de los clientes clave en el contexto actual.
  4. VentaTopAA: mismo filtro de clientes + desplazamiento de fechas al año anterior (dos filtros que se combinan).
  5. RETURN: variación interanual de los clientes clave.

2 · El "checklist de contexto" para predecir un resultado

✅ Hazte estas 5 preguntas en cada CALCULATE
  1. ¿Dónde estoy? ¿En qué celda/total se evalúa? ¿Qué slicers hay?
  2. ¿Hay contexto de fila? Si sí, ¿habrá transición que filtre cosas que no esperaba?
  3. Por cada filtro: ¿lo agrego, lo reemplazo o lo quito? ¿sobre qué columna?
  4. ¿Qué conservo? ¿Qué filtros del visual siguen vivos tras el CALCULATE?
  5. ¿El total cuadra? Mentalmente evalúa también la fila Total (sin filtro de desglose); muchos bugs solo aparecen ahí.

5 · Identificar errores de contexto típicos

❌ El "filtro fantasma" de la transición

Dentro de un iterador, una medida arrastra todas las columnas de la fila como filtro. Si iteras una tabla con muchas columnas, puedes filtrar por cosas que no querías. Solución: itera sobre VALUES(Dim[Clave]) (solo la clave), no sobre la tabla entera.

❌ Reemplazo no intencionado

Pasar Calendario[Año]=2025 dentro de una medida que el usuario filtró por 2026 reemplaza su selección sin avisar. Si querías intersectar y no reemplazar, usa KEEPFILTERS(Calendario[Año]=2025).

6 · Anticipar el resultado: ejercicio final del módulo

✏️ Lee y predice (nivel senior)

Contexto de celda: {Año=2026, Región=Sur}. Sur en 2026 = 800; Sur en 2025 = 1000; todas las regiones 2026 = 5000.

DAXMix Var =
VAR A = CALCULATE ( [Ventas], SAMEPERIODLASTYEAR ( Calendario[Fecha] ) )
VAR B = CALCULATE ( [Ventas], ALL ( Clientes[Región] ) )
RETURN DIVIDE ( [Ventas], B ) - DIVIDE ( A, B )
Ver solución

A = Sur 2025 = 1000 (reemplaza fecha, conserva Región=Sur). B = quita filtro de Región pero conserva Año=2026 → 5000. [Ventas] = Sur 2026 = 800.
RETURN = 800/5000 − 1000/5000 = 0,16 − 0,20 = −0,04 (la cuota de Sur cayó 4 puntos respecto al peso que tendría con sus ventas del año anterior). Si predijiste el signo y las dos divisiones, ya lees DAX como senior.

🎭 Analogía: Depurar es como revisar una receta que salió mal

No tiras el plato entero: pruebas cada ingrediente. ¿La sal (el filtro de año)? ¿El fuego (la transición)? ¿Olvidaste un paso (un KEEPFILTERS)? El VAR + RETURN temporal es tu cuchara para probar cada parte por separado.

ALL vs ALLSELECTED vs ALLEXCEPT — el cuadro que resuelve la confusión ⚠️

  Contexto: slicer Año=2026 ; matriz con Clientes en filas ; estás en Cliente A

  [Ventas]                  → A en 2026                (todo el contexto)
  ALL(Clientes)             → TODOS los clientes, 2026  (quita Cliente, deja Año)
  ALLSELECTED(Clientes)     → clientes del slicer, 2026 (quita solo el desglose de fila)
  ALLEXCEPT(Clientes,[País])→ todos los de MISMO país, 2026 (conserva País)
  ALL(Ventas)               → TODO, ignora hasta el Año (peligroso)

KEEPFILTERS: intersección en vez de reemplazo ⚠️

DAX-- Sin KEEPFILTERS: reemplaza. Aunque el visual esté en "Premium",
-- esto muestra Estándar (sobrescribe el filtro de la celda)
CALCULATE ( [Ventas], Productos[Tipo] = "Estándar" )

-- Con KEEPFILTERS: intersecta. En la celda "Premium" da BLANK
-- (Premium ∩ Estándar = ∅); en la celda "Estándar" da su valor
CALCULATE ( [Ventas], KEEPFILTERS ( Productos[Tipo] = "Estándar" ) )
⚙️ Qué ocurre internamente: Por qué FILTER conserva y un predicado reemplaza

Un predicado simple Productos[Tipo]="X" equivale a FILTER(ALL(Productos[Tipo]), ...): el ALL implícito borra el filtro previo de esa columna → por eso reemplaza. KEEPFILTERS quita ese ALL implícito → por eso intersecta. Entender esto te da control fino sobre cada filtro.

Casos de negocio

DAX% sobre Total Compañía   = DIVIDE ( [Ventas], CALCULATE ( [Ventas], ALL ( Clientes ) ) )
% dentro de su País      = DIVIDE ( [Ventas], CALCULATE ( [Ventas], ALLEXCEPT ( Clientes, Clientes[País] ) ) )
% sobre lo Seleccionado  = DIVIDE ( [Ventas], CALCULATE ( [Ventas], ALLSELECTED ( Clientes ) ) )
❌ ALL borra más de lo que crees

ALL(Ventas) (sobre la tabla de hechos entera) ignora todos los filtros, incluido el año del slicer. Para denominadores casi siempre quieres ALL(Dimensión) o REMOVEFILTERS(columna concreta), no ALL(TablaHechos).

✅ Decisión rápida
  • ¿Total de toda la compañía ignorando slicers de esa dim? → ALL
  • ¿Total de lo que el usuario seleccionó en el slicer? → ALLSELECTED
  • ¿% dentro de una categoría padre? → ALLEXCEPT (conserva la categoría)
  • ¿Añadir condición sin pisar la del visual? → KEEPFILTERS
  • ¿Solo quiero que se lea claro "quitar filtros"? → REMOVEFILTERS

💡 Por qué importa: Cuando el cálculo vive dentro de la fila

Si necesitas multiplicar dos columnas, ponderar, o evaluar una medida por cada elemento de una dimensión, necesitas un iterador. La diferencia con SUM no es cosmética: SUM agrega una columna ya existente; SUMX calcula algo nuevo en cada fila y luego agrega.

FunciónHaceEjemplo
SUMXSuma una expresión por filaSUMX(Ventas, Ventas[Cant]*Ventas[Precio])
AVERAGEXPromedia una expresión por filaAVERAGEX(Clientes,[Ventas]) (media por cliente)
COUNTXCuenta filas donde la expresión no es vacíaCOUNTX(Ventas, IF(Ventas[Importe]>0,1))
RANKXAsigna ranking según una expresiónRANKX(ALL(Clientes),[Ventas])

El patrón clave: iterar dimensión + medida (transición)

DAX-- Media de facturación POR CLIENTE (no media de tickets)
Facturación Media x Cliente =
AVERAGEX ( VALUES ( Clientes[ClienteID] ), [Facturación] )
⚙️ Qué ocurre internamente

Itera la lista de clientes; [Facturación] dispara transición y da el total de cada cliente; AVERAGEX promedia esos totales. Muy distinto de AVERAGE(Ventas[Importe]), que promediaría importes de línea.

⚠️ Performance: itera lo pequeño

Iterar la tabla de hechos con una medida dentro (transición por millones de filas) es lento. Itera sobre la dimensión (pocas filas) siempre que puedas. Ver Performance.

❌ Olvidar que X necesita una tabla

El primer argumento de un iterador es una tabla: SUMX(Ventas, ...), no SUMX(Ventas[Importe], ...). Si pones una columna directamente, conceptualmente piensa "¿sobre qué tabla itero?".

✅ Mejores prácticas

Usa iteradores para: precio×cantidad, conversión de moneda fila a fila, ponderaciones, y cálculos "por entidad" (por cliente, por producto) combinados con transición.

Guardar resultados intermedios con nombre. Mejoran legibilidad, performance y depuración a la vez. La marca de un código profesional.

Importancia★★★★★Dificultad🟢 FundamentalUsoDiario

💡 Por qué importa: Tres beneficios en una sola herramienta
  • Legibilidad: nombras cada paso, la medida se lee como prosa.
  • Performance: una VAR se evalúa una sola vez y se reutiliza, en vez de recalcular la misma expresión varias veces.
  • Depuración: puedes devolver cualquier VAR temporalmente para inspeccionarla.
DAXVar % Ventas YoY =
VAR VentasActual = [Ventas]
VAR VentasAA     = CALCULATE ( [Ventas], SAMEPERIODLASTYEAR ( Calendario[Fecha] ) )
VAR Diferencia   = VentasActual - VentasAA
RETURN
    DIVIDE ( Diferencia, VentasAA )
⚠️ Las VAR congelan el contexto donde se definen ⚠️

Una VAR se evalúa en el momento y contexto de su declaración, no donde se usa. Si defines VAR x = [Ventas] fuera de un CALCULATE y luego usas x dentro de ese CALCULATE, x conserva el valor original, no el modificado. Esto es una herramienta poderosa (capturar un valor antes de cambiar el contexto), pero sorprende a quien no lo sabe.

DAXEjemplo =
VAR VentaCelda = [Ventas]                    -- congela ventas de ESTA celda
RETURN
    CALCULATE ( VentaCelda, ALL ( Clientes ) ) -- sigue siendo el valor de la celda,
                                              -- ALL no lo afecta porque ya está calculado
❌ Recalcular en vez de guardar

Repetir CALCULATE([Ventas], SAMEPERIODLASTYEAR(...)) tres veces en una medida la hace más lenta y frágil. Guárdalo en una VAR y reúsalo.

✅ Mejores prácticas
  • Una VAR por cada concepto con nombre de negocio (VentasAA, Umbral), no nombres como x1.
  • Capturar un valor antes de un CALCULATE es un patrón legítimo y útil (sim. de cliente individual).
  • No puedes asignar una VAR a otra después; son inmutables. Encadena con varias VAR.

RANKX — posición dentro de un universo

DAXRanking Ventas =
IF (
    HASONEVALUE ( Clientes[ClienteID] ),
    RANKX (
        ALLSELECTED ( Clientes[ClienteID] ),  -- universo: clientes seleccionados
        [Ventas],                              -- criterio
        ,                                      -- valor (opcional)
        DESC,                                  -- mayor primero
        DENSE                                  -- sin saltos en empates
    )
)
⚙️ Qué ocurre internamente

RANKX evalúa [Ventas] para cada elemento del universo, los ordena y devuelve la posición del elemento actual. El universo (1er argumento) lo es todo: ALL rankea contra todos, ALLSELECTED contra lo filtrado. El guard HASONEVALUE evita un ranking absurdo en la fila Total.

TOPN — quedarse con los N mejores (devuelve tabla)

DAXVentas Top 5 Clientes =
CALCULATE (
    [Ventas],
    TOPN ( 5, ALLSELECTED ( Clientes[ClienteID] ), [Ventas], DESC )
)
DAX-- "Resto" (no-top): total menos top
Ventas Cola = [Ventas Totales] - [Ventas Top 5 Clientes]
⚠️ Empates

En RANKX, DENSE no salta posiciones tras empate (1,2,2,3); el modo por defecto sí (1,2,2,4). En TOPN, si hay empate en el puesto N, devuelve todos los empatados: "Top 5" puede dar 6 filas. Tenlo en cuenta en KPIs de concentración.

❌ Rankear sin fijar el universo

RANKX(Clientes, ...) sin ALL/ALLSELECTED rankea dentro del contexto de cada fila → casi siempre da 1 en todas. Debes liberar el filtro de la columna que rankeas.

✅ Mejores prácticas

Para análisis de concentración (Pareto 80/20, top clientes/productos, ABC) combina TOPN con medidas de "resto". Para tablas de clasificación, RANKX con guard de total. Ver Casos de negocio.

Ventas YTD (AF) = TOTALYTD ( [Ventas], Calendario[Fecha], "06-30" )   -- año fiscal cierra 30-jun
Ventas AA       = CALCULATE ( [Ventas], SAMEPERIODLASTYEAR ( Calendario[Fecha] ) )

Media móvil (rolling) con DATESINPERIOD

DAXVentas Móvil 12M =
CALCULATE (
    [Ventas],
    DATESINPERIOD ( Calendario[Fecha], MAX ( Calendario[Fecha] ), -12, MONTH )
)
Promedio Móvil 12M = DIVIDE ( [Ventas Móvil 12M], 12 )
⚙️ Qué ocurre internamente: Cómo encaja con CALCULATE

DATESINPERIOD(...,-12,MONTH) devuelve las fechas de los últimos 12 meses hasta la última visible. CALCULATE reemplaza el filtro de fecha por ese rango y conserva el resto (producto, región). Por eso el rolling respeta tus otros slicers.

❌ DATEADD necesita una columna de fechas, no una medida

DATEADD([Fecha visible], ...) no funciona: el primer argumento es la columna Calendario[Fecha]. Además, DATEADD necesita fechas contiguas en el contexto; con selecciones irregulares usa SAMEPERIODLASTYEAR o revisa el calendario.

❌ YoY que no respeta el año fiscal

Si tu cierre no es 31-dic, DATESYTD/TOTALYTD sin el parámetro de fin de año fiscal dan acumulados incorrectos. Pasa "06-30" (o el que corresponda) y marca bien la tabla de fechas.

✅ Mejores prácticas
  • Crea medidas base reutilizables: [Ventas] → [Ventas YTD] → [Var YoY], encadenadas.
  • Usa DATEADD para comparativos flexibles; SAMEPERIODLASTYEAR para el YoY estándar (más legible).
  • Para rolling y forecast móvil, DATESINPERIOD. Ver Forecast corporativo.

El concepto central: la tabla desconectada ⭐

🎭 Analogía: El mando a distancia

Una tabla desconectada (sin relación con el modelo) es como un mando a distancia: sus botones (los valores del slicer) no están "cableados" a los datos. No filtran nada por sí solos. Es tu medida la que lee qué botón pulsaste (SELECTEDVALUE) y decide qué hacer con esa señal. Si la tabla estuviera relacionada, filtraría datos en vez de darte un parámetro libre.

⚙️ Qué ocurre internamente: Por qué desconectada y no relacionada
  • Una tabla relacionada filtra los hechos: seleccionar "+10%" intentaría mostrar solo filas "+10%" (que no existen) → BLANK.
  • Una tabla desconectada no filtra nada: solo expone un valor que tú lees y usas como multiplicador/parámetro. El cálculo lo haces tú en DAX.

Parte 0 · Tres palabras que necesitas entender primero

🎭 Analogía: El slicer, el parámetro y la tabla "suelta"
  • Slicer (segmentación): una caja con botones o un control deslizante en el informe. Al pulsarlo, filtra lo que se ve. Es el "mando" con el que el usuario juega.
  • Parámetro What-If: un asistente de Power BI que, con un par de clics, te crea automáticamente un slicer de porcentajes (por ejemplo de −30% a +30%). Es la forma más fácil de tener un slicer numérico sin escribir nada.
  • Tabla "suelta" (desconectada): una tabla que no está unida con flechas al resto del modelo. Sirve para que el slicer te dé un valor sin filtrar los datos. Si estuviera unida, al elegir "+10%" intentaría buscar ventas "del +10%" (que no existen) y saldría vacío.

Idea clave: el slicer te entrega un número; tú, con una fórmula, decides qué hacer con ese número. Power BI no "ajusta solo": tú escribes una medida que multiplica las ventas por ese número.

Parte 2 · MODO 1 — Ajustar TODOS los clientes por igual 🟢

Objetivo: un slicer de % que sube o baja las ventas de todos a la vez.

✅ Paso a paso en Power BI
  1. Ve a la cinta Modelado → Nuevo parámetro → Rango numérico (en versiones antiguas: Modelado → Nuevo parámetro).
  2. Rellena la ventana así:
    • Nombre: Ajuste %
    • Tipo de datos: Número decimal
    • Mínimo: -0,30   Máximo: 0,30   Incremento: 0,05   Predeterminado: 0
  3. Deja marcada la casilla "Agregar segmentación de datos a esta página" y pulsa Crear.
  4. Power BI hace tres cosas solo: crea una tabla suelta llamada Ajuste %, una medida Ajuste % Valor, y coloca el slicer en el lienzo. 🎉
⚙️ Qué ocurre internamente: Qué te creó Power BI por dentro (no tienes que escribirlo, ya está)
DAX-- La tabla suelta de valores del slicer:
Ajuste % = GENERATESERIES ( -0.30, 0.30, 0.05 )

-- La medida que LEE el botón pulsado en el slicer:
Ajuste % Valor = SELECTEDVALUE ( 'Ajuste %'[Ajuste %], 0 )

En palabras: SELECTEDVALUE mira qué número está elegido en el slicer. Si eliges +10% devuelve 0,10; si no eliges nada, devuelve 0 (sin cambio).

Ahora la única medida que escribes tú. Inicio → Nueva medida:

DAXVentas Ajustadas =
[Ventas Base] * ( 1 + [Ajuste % Valor] )
⚙️ Qué ocurre internamente: Léelo en palabras normales

"Toma las ventas base y multiplícalas por (1 + el porcentaje del slicer)". Si el slicer está en +10% (0,10): Ventas × (1 + 0,10) = Ventas × 1,10 → suben un 10%. Si está en −20%: × 0,80 → bajan un 20%. Como la fórmula no nombra a ningún cliente, el mismo % se aplica a todos por igual: en la fila de cada cliente multiplica las ventas de ese cliente; en el total, el total.

✅ Cómo probarlo
  1. Pon una tabla (visual) con Clientes[Cliente], [Ventas Base] y [Ventas Ajustadas].
  2. Mueve el slicer Ajuste %. Verás que la columna "Ventas Ajustadas" cambia para todos los clientes a la vez. ✔

Parte 3 · MODO 2 — Ajustar UN SOLO cliente (con dos slicers) 🟡

Objetivo: un slicer para elegir qué cliente y otro para el %. Solo ese cliente cambia; el resto queda igual.

✅ Paso 1 · Crear el slicer de "cliente objetivo" (una lista de clientes SUELTA)
  1. Cinta Modelado → Nueva tabla.
  2. Escribe: 
    DAXCliente Objetivo = DISTINCT ( Clientes[Cliente] )
    Esto crea una copia suelta de la lista de clientes (Power BI no la une con flechas: justo lo que queremos).
  3. Inserta un slicer nuevo y arrástrale el campo Cliente Objetivo[Cliente]. Este será el slicer de "a quién ajustar".
⚠️ ¿Por qué una copia SUELTA y no la lista normal de clientes?

Si usaras el slicer con la tabla Clientes normal (la que está unida), al elegir "Cliente B" desaparecerían A y C de la tabla: filtrarías los datos. Con la copia suelta, el slicer solo te dice "el objetivo es B" sin esconder a los demás. Así puedes ver a todos y ajustar solo a uno.

Paso 2 · Reutiliza el slicer Ajuste % del Modo 1 (el del porcentaje). Paso 3 · Reescribe la medida:

DAXVentas Ajustadas =
VAR Pct = [Ajuste % Valor]                       -- el % del slicer
RETURN
    SUMX (
        VALUES ( Clientes[Cliente] ),            -- recorre cliente por cliente
        VAR Base = [Ventas Base]                 -- ventas de ESTE cliente
        VAR EsObjetivo =                         -- ¿este cliente es el elegido?
            Clientes[Cliente] IN VALUES ( 'Cliente Objetivo'[Cliente] )
        RETURN
            IF ( EsObjetivo, Base * ( 1 + Pct ), Base )  -- sí: ajusta; no: igual
    )
⚙️ Qué ocurre internamente: Léelo en palabras normales, línea por línea
  1. Pct = el porcentaje que pusiste en el slicer.
  2. SUMX(VALUES(Clientes[Cliente]), …) = "ve cliente por cliente y al final suma todo".
  3. Base = las ventas de ese cliente concreto (la fila en la que estamos).
  4. EsObjetivo = pregunta "¿el cliente de esta fila está entre los elegidos en el slicer de objetivo?". Da Verdadero o Falso.
  5. IF(...) = si es el objetivo, multiplica por (1 + %); si no, deja sus ventas tal cual.

Resultado: eliges "Cliente B" y "+20%" → solo B sube un 20%; A y C no se tocan. Y como usamos SUMX, el total también sale bien (suma a B ya ajustado con los demás sin tocar).

✅ Truco útil

Si no eliges ningún cliente en el slicer de objetivo, VALUES('Cliente Objetivo'[Cliente]) contiene a todos → se comporta como el Modo 1 (ajuste a todos). Una sola medida te sirve para los dos modos.

Parte 4 · MODO 3 — Un cliente con un %, el resto con otro % 🔴

Objetivo: el cliente (o clientes) que elijas reciben un %, y todos los demás reciben otro % distinto. Todo con slicers.

✅ Paso 1 · Crea DOS parámetros What-If de porcentaje
  1. Modelado → Nuevo parámetro → Rango numérico, nómbralo Ajuste Objetivo % (mín −0,30, máx 0,30, incremento 0,05). Marca "agregar segmentación". Crear.
  2. Repite y nómbralo Ajuste Resto %. Ya tienes dos slicers de %: uno para el cliente objetivo y otro para el resto.
  3. Mantén el slicer Cliente Objetivo del Modo 2 para elegir a quién.

Paso 2 · La medida que combina ambos:

DAXVentas Ajustadas =
VAR PctObjetivo = [Ajuste Objetivo % Valor]      -- % para el/los elegidos
VAR PctResto    = [Ajuste Resto % Valor]         -- % para todos los demás
RETURN
    SUMX (
        VALUES ( Clientes[Cliente] ),
        VAR Base = [Ventas Base]
        VAR EsObjetivo =
            Clientes[Cliente] IN VALUES ( 'Cliente Objetivo'[Cliente] )
        VAR PctAplicado = IF ( EsObjetivo, PctObjetivo, PctResto )  -- elige el % correcto
        RETURN
            Base * ( 1 + PctAplicado )
    )
⚙️ Qué ocurre internamente: Qué hace, en palabras

Por cada cliente decide: "¿es de los elegidos? entonces uso el % objetivo; si no, uso el % del resto". Luego multiplica sus ventas por (1 + ese %). Así, con tres slicers (cliente, % objetivo, % resto) puedes decir, por ejemplo: "Cliente B sube +25%, y todos los demás bajan −5%". Cambias los slicers y todo el informe se recalcula al instante.

Lo que quieresCómo dejas los slicers
Todos igual (+10%)No elijas cliente · "Resto %" = +10% (o usa el Modo 1)
Solo Cliente B +20%Cliente = B · "Objetivo %" = +20% · "Resto %" = 0%
B sube +25% y el resto baja −5%Cliente = B · "Objetivo %" = +25% · "Resto %" = −5%
B y D suben +15%, el resto 0%Cliente = B y D · "Objetivo %" = +15% · "Resto %" = 0%
⚠️ "Un cliente con %A y OTRO cliente con %B distinto, a la vez"

Con slicers, el patrón natural es "objetivo vs resto" (los elegidos comparten un %, los demás otro). Si necesitas que cada cliente tenga un % diferente y simultáneo (B +25%, C −15%, D +40%… todos a la vez), un slicer se queda corto porque solo guarda un valor a la vez. Dos caminos honestos:

  • Por segmentos, con estos mismos slicers: ajustas un grupo (objetivo) frente al resto; suele cubrir el 90% de las necesidades reales de planeamiento.
  • Cada cliente con su % distinto a la vez: tienes dos caminos, explicados paso a paso en el Escenario 3 → (1) una tabla de factores (un % por cliente; ideal para muchos clientes) y (2) un slicer por cliente (barras en vivo; ideal para pocos), más un híbrido que combina ambos. Ahí tienes los clics y las medidas listas para copiar.

DAXAjustes =
DATATABLE (
    "Etiqueta", STRING,
    "Factor",   DOUBLE,
    {
        { "-30%", 0.70 }, { "-20%", 0.80 }, { "-10%", 0.90 }, { "-5%", 0.95 },
        { "+5%", 1.05 }, { "+10%", 1.10 }, { "+20%", 1.20 }, { "+30%", 1.30 }
    }
)
⚠️ NO la relaciones con nada

Ajustes queda flotando, sin relación. Si la conectas, romperás el simulador. El slicer usará Ajustes[Etiqueta]; el cálculo usará Ajustes[Factor].

Paso 2 · Leer la selección

DAXFactor Ajuste = SELECTEDVALUE ( Ajustes[Factor], 1 )   -- 1 = sin cambio si no se elige nada

Paso 3 · La medida del simulador

DAXVenta Ajustada =
SUMX (
    VALUES ( Proyeccion[Cliente] ),
    CALCULATE ( SUM ( Proyeccion[VentaProyectada] ) ) * [Factor Ajuste]
)

O, si solo necesitas el total agregado, la versión simple:

DAXVenta Ajustada (simple) = SUM ( Proyeccion[VentaProyectada] ) * [Factor Ajuste]
⚙️ Qué ocurre internamente: Qué ocurre en cada paso al elegir "+10%"
  1. El slicer pone en el contexto Ajustes[Etiqueta]="+10%". Como la tabla está desconectada, esto NO filtra Proyeccion.
  2. SELECTEDVALUE(Ajustes[Factor],1) ve que hay un único valor visible → devuelve 1,10.
  3. SUM(Proyeccion[VentaProyectada]) se evalúa con el contexto normal (todos los clientes, o el que filtren otros slicers) → la suma base.
  4. Se multiplica por 1,10 → todas las ventas suben un 10%. En la versión SUMX, esto pasa cliente a cliente y luego se acumula (idéntico resultado para un ajuste uniforme, pero preparado para el Escenario 4).

Por qué funciona "para todos" automáticamente

No hay nada que diga "cliente A" o "cliente B": el factor multiplica la base sea cual sea el conjunto de clientes en el contexto. En la fila de cada cliente de una matriz, la base es la de ese cliente; en el total, es el total. El mismo factor aplica a todos. Ese es el sentido de "ajuste global".

❌ Olvidar el valor por defecto en SELECTEDVALUE

Sin el segundo argumento (1), cuando no hay selección o hay varias, SELECTEDVALUE devuelve BLANK y base * BLANK = BLANK: el informe se vacía. Pon siempre el default 1 (neutro multiplicativo).

✅ Mejores prácticas

Guarda el factor en una medida ([Factor Ajuste]) y reúsala en todas las medidas dependientes (ventas, margen, EBITDA). Cambias la lógica del slicer en un solo lugar.

✏️ Ejercicio

Base total = 1.000. Eliges "−20%". ¿Qué muestra [Venta Ajustada] en el total y en un cliente cuya base es 250?

Solución

Factor = 0,80. Total: 1.000×0,80 = 800. Cliente: 250×0,80 = 200. El mismo factor, distinta base por contexto.

El problema y por qué es más difícil

Ahora el ajuste NO es uniforme: solo Cliente A sube 20%, B y C quedan igual. La medida debe, fila a fila, preguntarse "¿soy el cliente objetivo? si sí, ajusto; si no, dejo la base".

Paso 1 · Slicers desconectados: cliente objetivo + factor

DAX-- Slicer 1: qué cliente ajustar (tabla desconectada con la lista de clientes)
ClienteObjetivo = DISTINCT ( Proyeccion[Cliente] )   -- copia desconectada (sin relación)
-- Slicer 2: el factor → reutilizamos la tabla Ajustes del Escenario 1
⚠️ Copia desconectada, no la dimensión real

Necesitas una segunda tabla de clientes desconectada para el slicer objetivo. Si usaras la dimensión real conectada, al elegir "A" filtrarías el modelo a A y perderías B y C de la vista. La copia desconectada solo te dice "a quién apuntar", sin filtrar.

Paso 2 · La medida: detectar el contexto de columna

DAXVenta Ajuste Individual =
VAR Objetivo   = SELECTEDVALUE ( ClienteObjetivo[Cliente] )   -- a quién ajustar
VAR Factor     = SELECTEDVALUE ( Ajustes[Factor], 1 )         -- cuánto
VAR ClienteFila = SELECTEDVALUE ( Proyeccion[Cliente] )       -- en qué fila estoy
VAR Base       = SUM ( Proyeccion[VentaProyectada] )
RETURN
    IF (
        ClienteFila = Objetivo,   -- ¿esta fila es el cliente objetivo?
        Base * Factor,            -- sí → ajusta
        Base                      -- no → base intacta
    )
⚙️ Qué ocurre internamente: Cómo identifica Power BI el cliente de cada celda
  1. En la fila "Cliente A" de la matriz, el contexto de filtro incluye Proyeccion[Cliente]="A".
  2. SELECTEDVALUE(Proyeccion[Cliente]) lee ese contexto → ClienteFila = "A".
  3. Objetivo viene del slicer desconectado → digamos "A".
  4. Como "A" = "A" → aplica Base * Factor. En la fila B, ClienteFila="B" ≠ "A" → devuelve Base. ✔

La clave es que SELECTEDVALUE(Proyeccion[Cliente]) "lee la fila" porque cada celda de la matriz impone su propio cliente en el contexto de filtro.

El problema del Total ⚠️

⚠️ En el total, ClienteFila no es único

En la fila Total hay varios clientes visibles → SELECTEDVALUE(Proyeccion[Cliente]) devuelve BLANK → nunca coincide con Objetivo → el total NO incluye el ajuste. El total quedaría mal. Solución: calcular el total como suma de las filas ajustadas con SUMX:

DAXVenta Ajuste Individual (total correcto) =
VAR Objetivo = SELECTEDVALUE ( ClienteObjetivo[Cliente] )
VAR Factor   = SELECTEDVALUE ( Ajustes[Factor], 1 )
RETURN
    SUMX (
        VALUES ( Proyeccion[Cliente] ),
        VAR ClienteFila = SELECTEDVALUE ( Proyeccion[Cliente] )
        VAR Base = CALCULATE ( SUM ( Proyeccion[VentaProyectada] ) )
        RETURN IF ( ClienteFila = Objetivo, Base * Factor, Base )
    )

Ahora SUMX recorre cliente a cliente (incluso en el total), aplica el IF por cliente y suma. El total cuadra con las filas.

❌ Usar la dimensión conectada para el slicer objetivo

Al seleccionar el cliente en un slicer conectado, filtras el modelo y desaparecen los demás clientes. Por eso el slicer objetivo debe salir de una tabla desconectada.

✅ Patrón senior

Captura con VAR las tres señales (objetivo, factor, fila), compáralas con IF, y envuelve en SUMX(VALUES(dim)) para que el total funcione. Esta estructura se reutiliza en los escenarios 3, 4 y 5.

━━━ MÉTODO 1 · Tabla de factores (la más escalable) ━━━

🎭 Analogía: Qué es una "tabla de factores"

Es simplemente una tablita con dos columnas: el cliente y su porcentaje. Una fila por cliente. Power BI la lee y le da a cada cliente el % que le corresponde. Piensa en ella como una lista de instrucciones: "a A súbele 10%, a B 25%, a C bájale 15%".

✅ Paso 1 · Crear la tabla (la forma más fácil: "Escribir datos")
  1. Cinta Inicio → Escribir datos (en inglés "Enter data").
  2. Se abre una mini-hoja. Crea dos columnas: Cliente y Ajuste %.
  3. Escribe una fila por cliente:
    ClienteAjuste %
    A0,10
    B0,25
    C-0,15
    (0,10 = +10%. Usa decimales: 25% se escribe 0,25.)
  4. Ponle de nombre Factores Cliente y pulsa Cargar. ¡Ya tienes la tabla en el modelo!

Alternativa profesional: en vez de "Escribir datos", conecta un Excel donde Planeamiento escribe los % y, al pulsar Actualizar, se recargan. Mismo resultado, pero el equipo mantiene el Excel.

✅ Paso 2 · La medida que reparte cada % a su cliente
DAXVenta Ajustada =
SUMX (
    VALUES ( Clientes[Cliente] ),                       -- ve cliente por cliente
    VAR Cli  = SELECTEDVALUE ( Clientes[Cliente] )       -- quién es este cliente
    VAR Base = [Ventas Base]                             -- sus ventas
    VAR Pct  =                                           -- busca SU % en la tabla
        LOOKUPVALUE (
            'Factores Cliente'[Ajuste %],
            'Factores Cliente'[Cliente], Cli,
            0                                            -- si no está en la tabla: 0%
        )
    RETURN Base * ( 1 + Pct )
)
⚙️ Qué ocurre internamente: Léelo en palabras
  1. SUMX(VALUES(Clientes[Cliente]), …) = recorre los clientes uno a uno y al final suma.
  2. Cli = el cliente de la fila actual (A, luego B, luego C…).
  3. LOOKUPVALUE(…) = "ve a la tabla Factores Cliente y tráeme el % de ESTE cliente". Es como un BUSCARV. Si el cliente no está, devuelve 0 (no se ajusta).
  4. Base × (1 + Pct) = aplica su porcentaje. A → ×1,10, B → ×1,25, C → ×0,85.

Con LOOKUPVALUE ni siquiera hace falta unir la tabla con flechas: la búsqueda es por nombre. Más simple para empezar.

✅ Paso 3 · Usarla
  1. Pon una matriz/tabla: Clientes[Cliente] en filas, [Ventas Base] y [Venta Ajustada] en valores. Verás 110, 250, 255.
  2. ¿Cambió el plan? Edita los % (Inicio → Transformar datos → tabla Factores Cliente, o el Excel) y pulsa Actualizar. Todos los clientes se recalculan con sus nuevos %.

━━━ MÉTODO 2 · Slicers (un control por cliente) ━━━

Si quieres tocarlo en vivo con barras y tienes pocos clientes, dale a cada cliente su propio slicer de %. Cada uno se mueve independientemente y todos a la vez.

✅ Paso a paso
  1. Crea un parámetro What-If por cliente: Modelado → Nuevo parámetro → Rango numérico, nómbralos Ajuste A %, Ajuste B %, Ajuste C % (mín −0,30, máx 0,30, paso 0,05). Marca "agregar segmentación" en cada uno.
  2. Tendrás una barra por cliente en la página.
  3. Usa esta medida (la misma idea del Escenario 4): 
    DAXVenta Ajustada =
SUMX (
    VALUES ( Clientes[Cliente] ),
    VAR Base = [Ventas Base]
    VAR Pct  =
        SWITCH (
            SELECTEDVALUE ( Clientes[Cliente] ),
            "A", [Ajuste A % Valor],     -- la barra de A
            "B", [Ajuste B % Valor],     -- la barra de B
            "C", [Ajuste C % Valor],     -- la barra de C
            0
        )
    RETURN Base * ( 1 + Pct )
)
  4. Mueve cada barra: el total de su cliente cambia al instante, cada uno con su propio %. ✔
⚠️ Esto es lo que un slicer SÍ puede hacer (y su límite)

Con una barra por cliente, cada cliente tiene su % distinto y simultáneo, en vivo. Funciona muy bien hasta ~5-8 clientes. Con 50 clientes serían 50 barras: inviable. Ahí el Método 1 (tabla) es el correcto. Regla simple: pocos y a mano → slicers; muchos → tabla de factores.

━━━ MÉTODO 3 · Híbrido (lo mejor de los dos) ⭐ ━━━

El patrón que más se usa en la práctica: la tabla de factores guarda el % "fino" de cada cliente, y un slicer global te deja jugar en vivo por encima de todos.

DAXVenta Ajustada =
VAR SliderGlobal = [Ajuste Global % Valor]               -- una barra para jugar en vivo
RETURN
    SUMX (
        VALUES ( Clientes[Cliente] ),
        VAR Cli  = SELECTEDVALUE ( Clientes[Cliente] )
        VAR Base = [Ventas Base]
        VAR PctCliente =
            LOOKUPVALUE ( 'Factores Cliente'[Ajuste %], 'Factores Cliente'[Cliente], Cli, 0 )
        RETURN Base * ( 1 + PctCliente ) * ( 1 + SliderGlobal )
    )
⚙️ Qué ocurre internamente

Cada cliente parte de su % de la tabla (granular, muchos clientes) y, encima, el slicer global escala a todos en vivo. Así combinas la escala de la tabla con la interactividad del slicer. Por ejemplo: la tabla pone los % por cuenta que negoció Comercial, y el director mueve el slicer global para ver "¿y si todo el mercado cae un 5% más?".

Tu situaciónMétodo recomendado
Muchos clientes, cada uno su %, se actualiza por Excel1 · Tabla de factores
Pocos clientes, quiero moverlos en vivo con barras2 · Slicers (uno por cliente)
% finos por cliente + jugar en vivo con un control3 · Híbrido
Un grupo con un % y el resto con otro"Objetivo vs resto" (ver Guía, Parte 4)
⚠️ La pieza invisible que hace que cada factor llegue a su cliente

En los tres métodos, lo que conecta cada % con su cliente es recorrer VALUES(Clientes[Cliente]) con SUMX: en cada vuelta "estamos parados" en un cliente y leemos solo su %. Es la transición de contexto. Por eso el total también sale bien (suma cada cliente ya ajustado).

💡 Por qué importa: La técnica clave: un slicer (parámetro What-If) por cliente

Como cada cliente necesita su propio porcentaje al mismo tiempo, y un slicer solo guarda un valor, creamos un parámetro What-If por cliente (A, B, C). Cada uno aparece como su propia barra deslizante. La medida luego "enruta" a cada cliente hacia su slicer. Esto funciona perfecto cuando los clientes son pocos y conocidos (como aquí). Para decenas de clientes, ver la nota de escalabilidad al final.

Paso 1 · Crear los slicers (clic a clic)

✅ Mejores prácticas

Paso 2 · La medida que enruta cada cliente a su slicer

DAXVenta Ajustada =
VAR Global = [Ajuste Global % Valor]                 -- la barra global (una para todos)
RETURN
    SUMX (
        VALUES ( Clientes[Cliente] ),                -- recorre cliente por cliente
        VAR Base = [Ventas Base]                     -- ventas de ESTE cliente
        VAR Indiv =                                  -- elige la barra de ESTE cliente
            SWITCH (
                SELECTEDVALUE ( Clientes[Cliente] ),
                "A", [Ajuste A % Valor],
                "B", [Ajuste B % Valor],
                "C", [Ajuste C % Valor],
                0                                    -- si aparece otro cliente: sin ajuste
            )
        RETURN
            Base * ( 1 + Global ) * ( 1 + Indiv )    -- base × global × individual
    )
⚙️ Qué ocurre internamente: Léelo en palabras, línea por línea
  1. Global = el % de la barra global (vale para todos).
  2. SUMX(VALUES(Clientes[Cliente]), …) = ve cliente por cliente y al final suma.
  3. Base = ventas del cliente de la fila actual.
  4. SWITCH(SELECTEDVALUE(Clientes[Cliente]), …) = mira QUIÉN es este cliente y devuelve el % de SU barra: si es "A" toma la barra de A, si es "B" la de B, etc.
  5. Base × (1+Global) × (1+Indiv) = aplica primero el % global y encima el individual.

Como cada fila de la matriz "es" un cliente, SELECTEDVALUE(Clientes[Cliente]) sabe en qué cliente está y le da el % de su propia barra. Por eso cada total de cliente queda afectado por un porcentaje distinto.

Paso 3 · Armar la visualización

✅ Mejores prácticas
  1. Inserta una matriz (o una tabla). En Filas pon Clientes[Cliente].
  2. En Valores pon [Ventas Base] y [Venta Ajustada].
  3. Coloca al lado las 4 barras de slicer. Muévelas: cada total de cliente cambia según su propia barra, y todos a la vez según la global. ✔
⚠️ ¿Por qué multiplicar y no sumar los % ?

"+10% global y +20% individual" se aplican uno sobre otro: 1,10 × 1,20 = 1,32 (+32%), no 1,10 + 0,20. Multiplicar modela el crecimiento compuesto. Si tu negocio los define como aditivos, sería Base × (1 + Global + Indiv): acláralo con Finanzas, es una decisión de negocio.

❌ No sumes dos medidas que comparten la base (doble conteo)

Si calcularas "ajuste global" y "ajuste individual" por separado y los sumaras, la venta base entraría dos veces. Por eso todo va en una sola medida, con una sola base, multiplicando los dos factores dentro del mismo SUMX.

✅ Cómo escala esto
  • Pocos clientes y conocidos: un parámetro What-If por cliente + SWITCH (lo de aquí). Totalmente por slicers.
  • Muchos clientes: una barra por cliente es inviable. Ahí la herramienta correcta es una tabla de factores en el modelo (un % por cliente, traída de Excel/origen, no editada en el informe) o parámetros de campo. Ver Escenario 3.

⚙️ Qué ocurre internamente: Celda "Feb-26 / Cliente B"

  1. [Ventas Base] ya está filtrado a Feb-26 y a B → 220. No filtras el mes a mano: lo pone la fila.
  2. SELECTEDVALUE(Clientes[Cliente]) = "B" (lo pone la columna) → el SWITCH toma [Ajuste B % Valor].
  3. Resultado de la celda: 220 × (1+Global) × (1+%B).

Paso a paso para construir la matriz

✅ Mejores prácticas
  1. Asegúrate de tener los slicers del Escenario 4 en la página (global + A, B, C) y la medida Venta Ajustada.
  2. Inserta una Matriz.
  3. Filas → arrastra Calendario[Mes].
  4. Columnas → arrastra Clientes[Cliente]. (Aquí es donde los clientes quedan en columnas.)
  5. Valores → arrastra [Venta Ajustada].
  6. Mueve los slicers: cada columna de cliente reacciona a su propio %, en todos los meses. ✔

⚠️ Por qué los TOTALES salen bien (gracias a SUMX)

En la columna Total mes (por ejemplo el total de Ene) no hay un cliente único: están A, B y C juntos. SUMX(VALUES(Clientes[Cliente]),…) recorre los tres, le da a cada uno su %, y luego suma. No escribas Calendario[Mes]=... en la medida: el mes ya lo impone la fila de la matriz. Tu medida solo se ocupa del ajuste por cliente; el visual aporta las coordenadas.

Una única medida (la del Escenario 4) cubre filas o columnas, un cliente o todos, global o individual. Cambias dónde colocas los campos y qué slicers tocas; la lógica no se duplica.

💡 Por qué importa: Si entiendes este flujo, puedes construir cualquier simulador

Los ocho escenarios anteriores son variaciones de un mismo mecanismo. Esta página lo abre en canal. Cuando lo domines, no necesitarás recetas: deducirás la medida desde primeros principios.

1 · ¿Qué hace el slicer internamente?

⚙️ Qué ocurre internamente

Un slicer no ejecuta código. Lo único que hace es añadir un filtro al contexto de filtro de todos los visuales de la página. Slicer de Ajustes[Etiqueta] en "+10%" → añade Ajustes[Etiqueta]="+10%" al contexto. Como Ajustes está desconectada, ese filtro no se propaga a ninguna tabla de datos: queda "disponible" para que una medida lo lea. Si estuviera conectada, se propagaría por la relación y filtraría hechos.

2 · ¿Qué hace SELECTEDVALUE internamente?

⚙️ Qué ocurre internamente

SELECTEDVALUE(Ajustes[Factor], 1) mira la columna Ajustes[Factor] dentro del contexto de filtro actual. Como el slicer dejó visible una sola fila ("+10%" → Factor 1,10), hay un único valor → lo devuelve. Internamente es IF(HASONEVALUE(Ajustes[Factor]), VALUES(Ajustes[Factor]), 1). Si hubiera 0 o varias filas visibles, devuelve el default. Convierte "qué botón pulsó el usuario" en un número usable.

3 · ¿Qué hace CALCULATE internamente aquí?

⚙️ Qué ocurre internamente

En CALCULATE(SUM(Proyeccion[VentaProyectada])) dentro del SUMX, CALCULATE ejecuta la transición de contexto: toma el cliente de la iteración actual y lo convierte en filtro sobre Proyeccion. Así "Base" pasa de ser el total a ser la base de ese cliente concreto. Sin ese CALCULATE, cada iteración sumaría el total completo.

4 · Cómo se construye el contexto de filtro de una celda

  Celda "Mar-26 / Cliente B" en la matriz, slicers: Ajuste="+10%", Objetivo="B"

  CONTEXTO DE FILTRO de la celda =
     { Calendario[Mes] = "Mar-26" }     ← de la FILA del visual
   ∪ { Cliente[Cliente] = "B" }         ← de la COLUMNA del visual
   ∪ { Ajustes[Etiqueta] = "+10%" }     ← del SLICER (desconectado: no propaga)
   ∪ { ClienteObjetivo[Cliente] = "B" } ← del SLICER (desconectado: no propaga)

5 · Cómo se propagan los filtros

   { Mes="Mar-26" }  ──1:*──►  Proyeccion  (queda solo marzo)
   { Cliente="B" }   ──1:*──►  Proyeccion  (queda solo B)
   ──────────────────────────────────────────────────────
   Intersección: Proyeccion filtrada a (Mar-26 ∧ B) = la celda exacta
   Ajustes / ClienteObjetivo: NO propagan (desconectadas) → solo se leen

6 · Cómo Power BI sabe qué clientes modificar y cuáles no

⚙️ Qué ocurre internamente: El corazón de la decisión

Dentro del SUMX(VALUES(Cliente[Cliente])), en cada iteración:

  1. ClienteFila = SELECTEDVALUE(Cliente[Cliente]) → el cliente de esta iteración (ej. "B").
  2. Se compara con Objetivo (del slicer desconectado, "B").
  3. IF(ClienteFila = Objetivo, ...) decide: coincide → ajusta; no coincide → base intacta.

Power BI no "sabe" nada mágico: tú comparas la coordenada de la celda (que el visual impone) con el parámetro del slicer (que tú lees). La igualdad ClienteFila = Objetivo es toda la inteligencia. Los que no cumplen la condición conservan su base — por eso el resto no se modifica.

8 · Cómo interactúan slicers con matrices, gráficos y tablas

⚙️ Qué ocurre internamente

El mecanismo es idéntico en cualquier visual, porque todos generan contexto de filtro igual:

  • Matriz: fila × columna → contexto bidimensional (lo visto arriba).
  • Gráfico de barras: cada barra es una categoría del eje → un contexto por barra.
  • Tabla: cada fila → un contexto. Tarjeta: un único contexto (a veces sin desglose → cuidado con SELECTEDVALUE).
  • Slicer: añade su filtro a todos los visuales de la página por igual.

Por eso la misma medida de simulación funciona en todos a la vez: no depende del tipo de visual, sino del contexto de filtro que cada uno genera.

✅ La regla mental definitiva del simulador

El visual aporta las coordenadas (qué celda). El slicer desconectado aporta los parámetros (qué hacer). La medida compara coordenadas con parámetros y decide. CALCULATE/transición trae la base correcta de cada coordenada. Con esas cuatro frases puedes reconstruir cualquiera de los 9 escenarios desde cero.

⚙️ Qué ocurre internamente: La regla de oro de performance

El Storage Engine (SE) es rápido, paralelo y cacheable; el Formula Engine (FE) es secuencial y caro. Una medida lenta casi siempre es una que obliga al FE a iterar fila a fila (muchos "callbacks" al SE). Optimizar = empujar el trabajo al SE y reducir lo que hace el FE. Lo mides con DAX Studio (Server Timings: tiempo SE vs FE).

Optimización del modelo (mayor impacto)

✅ Checklist de modelo
  • Star schema: dimensiones pequeñas, hechos estrechos. Ver modelado.
  • Reduce cardinalidad: claves enteras, no GUIDs/textos; separa fecha y hora en dos columnas.
  • Elimina columnas que no usas: cada columna pesa en memoria, aunque no la muestres.
  • Evita columnas calculadas que dependan del contexto; prefiere medidas.
  • Tipos correctos: enteros para claves, decimales fijos para importes; evita flotantes innecesarios.
  • Una sola tabla de fechas marcada; desactiva "Auto fecha/hora" (crea jerarquías ocultas que inflan el modelo).

Optimización de medidas

✅ Mejores prácticas
  • Filtra con predicados simples antes que con FILTER; reserva FILTER para lógica multi-columna.
  • Evita la transición masiva: itera dimensiones, no hechos.
  • Mueve cálculos invariantes fuera de los iteradores (a una VAR previa).
  • Usa DIVIDE y guards (HASONEVALUE) para evitar trabajo y errores en totales.
  • Prefiere SUMMARIZE/SUMMARIZECOLUMNS bien hechos a iteraciones manuales para agrupar.
⚠️ Mide antes de optimizar

No optimices a ciegas. Usa el Performance Analyzer de Power BI para ver qué visual tarda, y DAX Studio (Server Timings) para separar tiempo de SE y FE. Optimizar la medida equivocada es tiempo perdido: el cuello suele ser un visual con demasiados elementos o un modelo mal comprimido.

✏️ Diagnóstico de performance

Una medida SUMX(Ventas, [Margen Cliente]) sobre 8M de filas tarda 15s. ¿Qué harías?

Solución

El problema es la transición por fila sobre 8M de registros (FE saturado). Reescribir para iterar la dimensión cliente (miles de filas), no los hechos: SUMX(VALUES(Clientes[ClienteID]), [Margen Cliente]). Si el margen es aditivo, mejor aún evitar el iterador y resolverlo con una agregación directa en el SE. Pasarías de 15s a milisegundos.

El modelo de P&L

Tabla de hechos Movimientos(Fecha, CuentaID, CECO, Escenario, Importe) + dimensión PlanCuentas con una columna Grupo (Ingresos, COGS, OPEX, D&A…) que define la estructura del P&L. La clasificación vive en la dimensión, no en la medida.

Bloques del EBITDA

DAXIngresos = CALCULATE ( SUM ( Movimientos[Importe] ), PlanCuentas[Grupo] = "Ingresos" )
COGS     = CALCULATE ( SUM ( Movimientos[Importe] ), PlanCuentas[Grupo] = "COGS" )
OPEX     = CALCULATE ( SUM ( Movimientos[Importe] ), PlanCuentas[Grupo] = "OPEX" )

Margen Bruto   = [Ingresos] - [COGS]
% Margen Bruto = DIVIDE ( [Margen Bruto], [Ingresos] )
EBITDA         = [Margen Bruto] - [OPEX]
% EBITDA       = DIVIDE ( [EBITDA], [Ingresos] )
⚙️ Qué ocurre internamente

Cada bloque reemplaza el filtro de Grupo y conserva el resto del contexto (mes, CECO, escenario). Por eso el mismo [EBITDA] funciona en cualquier celda: por mes, por unidad de negocio, por región — el contexto lo aporta el visual.

OPEX vs CAPEX: la distinción contable

⚠️ CAPEX no va al P&L directamente

El OPEX (gasto operativo) impacta el resultado del periodo. El CAPEX (inversión en activos) no es gasto del periodo: se capitaliza y golpea el P&L vía amortización (D&A) a lo largo de la vida útil. Modelar esto bien evita el error de "restar el CAPEX del EBITDA".

DAXCAPEX        = CALCULATE ( SUM ( Movimientos[Importe] ), PlanCuentas[Grupo] = "CAPEX" )
Amortización = CALCULATE ( SUM ( Movimientos[Importe] ), PlanCuentas[Grupo] = "D&A" )
EBIT         = [EBITDA] - [Amortización]    -- EBIT sí incluye D&A; EBITDA no

OPEX por naturaleza y por centro de coste

DAX% OPEX sobre Ingresos = DIVIDE ( [OPEX], [Ingresos] )
OPEX por CECO         = [OPEX]   -- en una matriz con CECO en filas, se desglosa solo
Peso CECO en OPEX     = DIVIDE ( [OPEX], CALCULATE ( [OPEX], ALL ( CentrosCoste ) ) )
✅ Mejores prácticas
  • Clasifica en la dimensión de cuentas (Grupo, Subgrupo, signo). Las medidas solo seleccionan grupos.
  • Define el signo de forma coherente (gastos negativos o positivos) y sé consistente en todo el modelo.
  • EBITDA = resultado antes de intereses, impuestos, depreciación y amortización: no restes D&A ni financieros.
✏️ Ejercicio

Ingresos 1.000, COGS 600, OPEX 250, D&A 50. ¿EBITDA, % EBITDA y EBIT?

Solución

Margen bruto = 400. EBITDA = 400 − 250 = 150. % EBITDA = 150/1000 = 15%. EBIT = 150 − 50 = 100. El D&A separa EBITDA de EBIT.

Aging de cuentas por cobrar

Pregunta de negocio: "¿cuánto me deben y desde hace cuánto?" El aging clasifica la deuda pendiente por tramos de antigüedad.

DAXDías Vencido =
VAR Hoy = MAX ( Calendario[Fecha] )    -- fecha de corte del análisis
RETURN
    SUMX (
        Facturas,
        VAR dias = DATEDIFF ( RELATED ( Facturas[FechaVencimiento] ), Hoy, DAY )
        RETURN dias
    )
DAXTramo Aging =
SWITCH (
    TRUE (),
    [Días Vencido] <= 0,  "Por vencer",
    [Días Vencido] <= 30, "1-30",
    [Días Vencido] <= 60, "31-60",
    [Días Vencido] <= 90, "61-90",
    "+90 (riesgo)"
)
Deuda Vencida +90 =
CALCULATE ( [Saldo Pendiente], FILTER ( Facturas, [Días Vencido] > 90 ) )
⚠️ El aging depende de la fecha de corte

El tramo cambia según el "hoy". Usa MAX(Calendario[Fecha]) como corte para que el aging respete el slicer de fecha y puedas ver fotos históricas. Mídelo por fecha de vencimiento, no de emisión → suele requerir USERELATIONSHIP.

Días de cobro (DSO) y cobranzas

DAXSaldo Pendiente = [Facturado] - [Cobrado]
DSO =
DIVIDE ( [Saldo Pendiente], [Facturado] ) * [Días del Periodo]   -- días de venta en la calle
% Cobrado = DIVIDE ( [Cobrado], [Facturado] )

Provisiones por incobrabilidad

DAXProvisión Incobrables =
SUMX (
    VALUES ( Aging[Tramo] ),
    VAR saldo = [Saldo Pendiente]
    VAR pct   = RELATED ( Aging[% Provisión] )   -- % de riesgo por tramo (ej. +90 = 50%)
    RETURN saldo * pct
)
⚙️ Qué ocurre internamente

Cada tramo de aging tiene un % de provisión (mayor cuanto más vencido). La provisión total es la suma ponderada del saldo por su riesgo. Es el patrón factor por categoría aplicado a riesgo crediticio.

Evolución de la deuda

DAXDeuda Inicio Periodo = CALCULATE ( [Saldo Pendiente], DATEADD ( Calendario[Fecha], -1, MONTH ) )
Variación Deuda      = [Saldo Pendiente] - [Deuda Inicio Periodo]
✅ Mejores prácticas
  • Aging y DSO siempre con fecha de corte explícita (MAX(Calendario[Fecha])).
  • Provisión por tramos en una tabla de parámetros (% por tramo), no hardcodeada.
  • Separa facturado y cobrado como medidas base; el saldo se deriva.

El patrón de escenarios en una sola tabla

Real, Presupuesto y Forecast conviven en la misma tabla de hechos, diferenciados por una dimensión Escenario. Una sola estructura, comparaciones triviales.

DAXReal     = CALCULATE ( SUM ( Movimientos[Importe] ), Escenario[Tipo] = "Real" )
Ppto     = CALCULATE ( SUM ( Movimientos[Importe] ), Escenario[Tipo] = "Presupuesto" )
Forecast = CALCULATE ( SUM ( Movimientos[Importe] ), Escenario[Tipo] = "Forecast" )

Análisis de variaciones

DAXDesvío Ppto        = [Real] - [Ppto]
Desvío Ppto %      = DIVIDE ( [Desvío Ppto], [Ppto] )
Cumplimiento Ppto  = DIVIDE ( [Real], [Ppto] )                 -- 1 = 100%
Desvío Forecast    = [Real] - [Forecast]
DAX-- Semáforo de desempeño (cuidado con el signo en cuentas de gasto)
Estado Presupuestario =
VAR cumpl = [Cumplimiento Ppto]
RETURN
    SWITCH (
        TRUE (),
        cumpl >= 0.98 && cumpl <= 1.02, "🟢 En línea",
        cumpl < 0.98,  "🔵 Por debajo",
        "🔴 Sobre presupuesto"
    )
⚠️ El signo del desvío depende de si es ingreso o gasto ⚠️

En ingresos, Real > Ppto es bueno. En gastos, Real > Ppto es malo. Un semáforo que no distingue cuenta dará verde a un sobrecoste. Solución: una columna Sentido en PlanCuentas (+1 ingreso / −1 gasto) y multiplicar el desvío por ella antes de evaluar el color.

Bridge de variación (puente Ppto → Real)

DAX-- Descomposición precio/volumen del desvío de ingresos
Efecto Volumen = ( [Unidades Real] - [Unidades Ppto] ) * [Precio Ppto]
Efecto Precio  = ( [Precio Real]  - [Precio Ppto] )  * [Unidades Real]
Check Bridge   = [Efecto Volumen] + [Efecto Precio] - [Desvío Ppto]   -- debe dar ≈ 0
⚙️ Qué ocurre internamente

El "puente" explica por qué el real difiere del presupuesto separando cuánto es por vender más/menos unidades (volumen) y cuánto por precio. Check Bridge ≈ 0 valida que la descomposición cuadra: una verificación que todo controller agradece.

Forecast = Real YTD + Presupuesto resto del año

DAXForecast Rolling =
VAR Corte = MAX ( Calendario[Fecha] )
VAR RealYTD =
    CALCULATE ( [Real], Calendario[Fecha] <= Corte )
VAR PptoResto =
    CALCULATE ( [Ppto], Calendario[Fecha] > Corte )
RETURN
    RealYTD + PptoResto
✅ Mejores prácticas
  • Un único modelo con dimensión Escenario: evita duplicar medidas por versión.
  • Maneja el signo de los desvíos por naturaleza de cuenta.
  • Acompaña el desvío con su descomposición (precio/volumen, mix) — es lo que pide la dirección.
  • Forecast rolling = real hasta el corte + plan después; el corte = MAX(Calendario[Fecha]) o un parámetro.

Concentración de clientes (Pareto 80/20)

DAXVentas Top 10 Clientes =
CALCULATE ( [Ventas], TOPN ( 10, ALLSELECTED ( Clientes[ClienteID] ), [Ventas], DESC ) )

% Concentración Top 10 =
DIVIDE ( [Ventas Top 10 Clientes], CALCULATE ( [Ventas], ALLSELECTED ( Clientes ) ) )
DAX-- Pareto acumulado: ¿qué % de clientes hace el 80% de las ventas?
Ventas Acumuladas % =
VAR VentaCliente = [Ventas]
VAR Acumulado =
    CALCULATE (
        [Ventas],
        FILTER ( ALLSELECTED ( Clientes[ClienteID] ), [Ventas] >= VentaCliente )
    )
RETURN
    DIVIDE ( Acumulado, CALCULATE ( [Ventas], ALLSELECTED ( Clientes ) ) )
⚙️ Qué ocurre internamente

El Pareto acumulado suma las ventas de todos los clientes con ventas ≥ las del actual (los más grandes) y lo divide por el total. Ordenando clientes por ventas, ves la curva de concentración: si el 20% superior da el 80%, tienes riesgo de dependencia.

Clasificación ABC

DAXClase ABC =
VAR p = [Ventas Acumuladas %]
RETURN SWITCH ( TRUE (), p <= 0.80, "A", p <= 0.95, "B", "C" )

Mix de producto y cross-sell

DAXMix Categoría %  = DIVIDE ( [Ventas], CALCULATE ( [Ventas], ALLSELECTED ( Productos[Categoría] ) ) )
Categorías x Cliente = AVERAGEX ( VALUES ( Clientes[ClienteID] ), DISTINCTCOUNT ( Ventas[CategoríaID] ) )
Ticket Medio     = DIVIDE ( [Ventas], DISTINCTCOUNT ( Ventas[FacturaID] ) )

Retención y clientes nuevos vs recurrentes

DAXClientes Activos = DISTINCTCOUNT ( Ventas[ClienteID] )

Clientes Nuevos =
VAR PrimeraCompra =
    CALCULATE ( MIN ( Ventas[Fecha] ), ALLEXCEPT ( Ventas, Ventas[ClienteID] ) )
RETURN
    COUNTROWS (
        FILTER (
            VALUES ( Ventas[ClienteID] ),
            CALCULATE ( MIN ( Ventas[Fecha] ) ) >= STARTOFMONTH ( MAX ( Calendario[Fecha] ) )
        )
    )

Clientes Recurrentes = [Clientes Activos] - [Clientes Nuevos]
Tasa Retención = DIVIDE ( [Clientes Recurrentes], CALCULATE ( [Clientes Activos], PREVIOUSMONTH ( Calendario[Fecha] ) ) )
⚠️ "Cliente nuevo" necesita una definición de negocio

¿Nuevo = primera compra en su vida, o primera en el periodo? Las medidas cambian por completo. Acuérdala con el área comercial antes de codificar; aquí "nuevo" = primera compra dentro del mes analizado.

✅ Mejores prácticas
  • Concentración y ABC con ALLSELECTED para respetar los slicers del usuario.
  • Combina con los simuladores: "¿qué pasa con el mix si subo precios al segmento A?"
  • Define con negocio los conceptos ambiguos (nuevo, activo, churn) y documéntalos en el modelo.
